## Design Philosophy

The backend is built with **Go** and follows a **Clean Architecture** approach:

- **Type Safety**: Using SQLC to generate type-safe Go code from raw SQL queries.
- **Layered Separation**: Clear boundaries between HTTP handlers, business logic (services), and data access (repositories).
- **Minimal Dependencies**: Leveraging the Go standard library where possible, with Echo as a lightweight web framework.
- **CGO-Free**: Using `modernc.org/sqlite` for a pure Go SQLite implementation, simplifying cross-compilation and deployment.
- **Single Container Deployment**: The entire application (API, database, and frontend) is delivered within a single Docker image, making it easy to deploy without external dependencies.
- **Static Asset Serving**: The backend binary serves the frontend assets from the filesystem at runtime.

## Data Access Layer (SQLC)

### 3.1 — Schema-First Development

We define our database structure in `api/sql/schema.sql` and our queries in `api/sql/queries.sql`. SQLC then generates type-safe Go code to interface with the database.

### 3.2 — Repository Pattern

The `repository` package wraps the generated SQLC code and provides a clean interface for the service layer. It also handles complex operations that aren't easily expressed in single SQL queries.

```go
// api/internal/repository/repository.go (Conceptual)

type Repository struct {
    *Queries // Generated by SQLC
    db      *sql.DB
}

func NewRepository(db *sql.DB) *Repository {
    return &Repository{
        Queries: New(db),
        db:      db,
    }
}
```

## Service Layer

Services contain the core business logic. They are responsible for:
- Orchestrating multiple repository calls.
- Handling complex validation.
- Managing side effects (e.g., image processing, cache invalidation).
- Providing a clean API for the HTTP layer.

Key services include:
- `AuthService`: Session management, password hashing (Argon2id).
- `PostService`: Markdown to HTML conversion, publishing workflows.
- `MediaService`: Image resizing, Exif extraction, file storage.
- `TagService`: Hierarchical tag management, post count recalculation.

## API Layer (Echo)

### 5.1 — Echo Framework

We use the [Echo](https://echo.labstack.com/) web framework for its performance and simplicity. It handles routing, middleware, and request/response binding.

### 5.2 — Handlers

Handlers are responsible for parsing request input, calling the appropriate service, and returning a JSON response.

```go
// Example Handler
func (h *PostHandler) GetPost(c echo.Context) error {
    id := c.Param("id")
    post, err := h.service.GetByID(c.Request().Context(), id)
    if err != nil {
        return err
    }
    return c.JSON(http.StatusOK, post)
}
```

### 5.3 — Middleware

- **AuthMiddleware**: Validates session cookies and populates user context.
- **Secure**: Adds standard security headers (CSP, HSTS, XSS protection).
- **CORS**: Configures allowed origins for frontend development.
- **Logger**: Standardized request logging.

## Security Model

### 6.1 — Authentication

- **Session-Based**: Uses HTTP-only, Secure, SameSite=Lax cookies.
- **Password Hashing**: Utilizes **Argon2id** for state-of-the-art password security.
- **Session Management**: Supports listing and revoking active sessions.

### 6.2 — Authorization

- **Admin Routes**: Protected by `AuthMiddleware`. Access to `/api/light/*` requires an active session.
- **Public Routes**: Some routes use `OptionalAuthMiddleware` to provide enhanced information for logged-in users while remaining accessible to the public.

### 6.3 — Content Security

- **Sanitization**: Markdown is rendered to HTML server-side using `goldmark`.
- **CSP**: A strict Content Security Policy is enforced via middleware to prevent XSS.

## Frontend Integration

The backend is designed to serve the frontend as a Single Page Application (SPA):

- **Static Asset Serving**: Echo serves the `frontend/` directory (mapped to `/assets/`).
- **SPA Fallback**: Any non-API request that doesn't match a static file is served the `index.html` file, allowing the frontend router to handle the path.
- **Cache Control**: Specific headers are set for `/assets/js/` and `/assets/css/` to prevent stale assets on mobile browsers (e.g., Safari on iOS).

## Media Processing

The `MediaService` handles the heavy lifting for uploaded assets:
- **Resizing**: Generates thumbnails and responsive image versions.
- **Exif**: Automatically extracts location and camera metadata from images.
- **Storage**: Files are stored in the `data/media` directory, organized by year/month.

---

## Background Tasks

The `SchedulerService` manages recurring and deferred tasks:
- **Scheduled Publishing**: Publishing posts at a future date.
- **System Maintenance**: Periodic cache cleaning and backup generation.
- **Image Processing**: Heavy image optimization tasks can be offloaded.

## Development Workflow

### Prerequisites
- Go 1.25+
- SQLC (for generating repository code)

### Running the API
```bash
cd api
go run cmd/api/main.go
```

### Database Migrations
Migrations are handled via SQL files in `api/sql/` and applied on startup.

### Generating Code
```bash
# Generate SQLC code
sqlc generate
```
